_Artikel ini memang sengaja dibuat sebagai dummy post untuk menguji tampilan blog, terutama saat tulisan teknikal berisi campuran **paragraf panjang**, `inline code`, tabel, dan banyak blok command._

Kalau sebuah blog teknikal hanya diuji dengan artikel pendek, biasanya masalah layout baru muncul belakangan: heading terlalu rapat, list bertumpuk, tabel terlalu sempit, atau code block terasa berbeda antar halaman. Karena itu artikel ini sengaja dibuat panjang dan berisi banyak variasi elemen agar kamu bisa melihat apakah hasil akhirnya benar-benar nyaman dibaca.

Dalam praktiknya, artikel seperti ini juga berguna sebagai _playground_ saat ingin menguji style tulisan baru, fitur copy button, dark mode, sampai bagaimana konten tampil di mobile. Kamu bisa anggap post ini sebagai halaman uji untuk seluruh komponen konten yang paling sering dipakai di blog operasional, dokumentasi internal, atau tutorial step-by-step.

## Skala Heading

Bagian ini tidak terlalu penting secara isi, tapi berguna untuk melihat ritme heading.

### Heading Level 3

Heading tingkat tiga biasanya cocok dipakai untuk membagi topik utama menjadi bagian yang masih cukup besar, misalnya instalasi, konfigurasi, verifikasi, dan troubleshooting.

#### Heading Level 4

Heading tingkat empat lebih cocok untuk langkah yang lebih kecil, seperti membuat direktori, menulis unit file, atau menjalankan verifikasi endpoint.

##### Heading Level 5

Biasanya level ini jarang dipakai di artikel normal, tetapi tetap perlu terlihat rapi kalau sewaktu-waktu muncul pada dokumentasi yang lebih kompleks.

###### Heading Level 6

Level paling kecil ini tidak sering dipakai, namun tetap berguna untuk menguji jarak antar heading dan paragraf.

## Paragraf Panjang

Sebagian besar artikel teknikal yang bagus sebenarnya tidak hanya berisi command. Yang lebih penting adalah konteks: mengapa langkah tertentu dilakukan, apa risikonya, apa hasil yang diharapkan, dan kapan pembaca sebaiknya berhenti sebelum melanjutkan ke langkah berikutnya. Kalau blog terlalu fokus pada blok code, pembaca bisa tahu _apa_ yang harus dijalankan, tetapi tidak benar-benar paham _kenapa_ command itu perlu dijalankan.

Di sisi lain, paragraf yang terlalu padat juga membuat artikel cepat melelahkan. Karena itu layout konten perlu menyeimbangkan tiga hal sekaligus: panjang baris yang nyaman, spacing antar paragraf yang cukup longgar, dan elemen visual seperti list atau blockquote yang bisa menjadi jeda alami di tengah bacaan. Artikel dummy ini sengaja memakai paragraf panjang agar kamu bisa mengecek apakah ritme baca di halaman post terasa konsisten dari atas sampai bawah.

Kadang penulis teknikal juga butuh membuat _manual line break_ untuk dua baris yang masih terkait langsung.  
Contoh ini sengaja memakai line break untuk melihat apakah hasilnya tetap natural dan tidak terasa janggal.

## Unordered List

- Item pertama menjelaskan konteks singkat.
- Item kedua menambahkan detail yang sedikit lebih teknis.
- Item ketiga berisi sub-item agar nesting list bisa diuji.
  - Sub-item A untuk catatan kecil.
  - Sub-item B untuk pengecualian.
  - Sub-item C dengan `inline code` di dalam list.
- Item keempat dipakai untuk melihat jarak akhir list terhadap paragraf sesudahnya.

Sesudah unordered list, paragraf ini dipakai untuk menguji apakah margin bawah dari list terlalu rapat atau terlalu jauh.

## Ordered List

1. Siapkan environment dan pastikan akses `sudo` tersedia.
2. Buat struktur direktori yang diperlukan.
3. Tulis file konfigurasi dengan nilai minimal yang aman.
4. Jalankan service dan cek status.
5. Verifikasi endpoint sebelum melanjutkan.

Kalau perlu langkah yang lebih detail, ordered list juga bisa memiliki nested list:

1. Instal dependency utama.
   1. Tambahkan repository.
   2. Jalankan update package.
   3. Instal binary utama.
2. Tulis konfigurasi.
   1. Buat file `.yml`.
   2. Isi target dan interval scrape.
3. Aktifkan service.

## Step-by-Step ala Docs

Kalau ingin gaya tutorial yang lebih dekat ke dokumentasi modern, pola `Steps` seperti ini biasanya lebih enak dibaca dibanding ordered list biasa karena tiap bagian terasa punya checkpoint yang jelas.

<Steps>
  <div>
    <h3>Step 1: Siapkan struktur direktori</h3>
    <p>Mulai dari membuat direktori konfigurasi dan file awal supaya langkah sesudahnya punya konteks yang rapi.</p>

```bash
sudo mkdir -p /etc/prometheus
sudo touch /etc/prometheus/prometheus.yml
sudo nano /etc/prometheus/prometheus.yml
```
  </div>

  <div>
    <h3>Step 2: Tulis konfigurasi minimal</h3>
    <p>Gunakan konfigurasi dasar lebih dulu agar service cepat hidup dan lebih mudah diverifikasi sebelum ditambah target lain.</p>

```yaml
global:
  scrape_interval: 15s

scrape_configs:
  - job_name: 'prometheus'
    static_configs:
      - targets:
          - localhost:9090
```
  </div>

  <div>
    <h3>Step 3: Jalankan service dan verifikasi</h3>
    <p>Setelah konfigurasi siap, aktifkan service lalu cek status untuk memastikan semua langkah sebelumnya benar.</p>

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now prometheus
sudo systemctl status prometheus
```
  </div>
</Steps>

## Elemen HTML yang Masih Relevan di Markdown

Markdown di blog teknikal sering dipakai berdampingan dengan HTML ringan untuk kasus tertentu.

<details>
<summary>Klik untuk membuka catatan tambahan</summary>

Kadang penulis butuh menyembunyikan bagian opsional seperti troubleshooting, alternatif command, atau catatan migrasi yang tidak selalu relevan untuk semua pembaca.

- Bisa berisi list
- Bisa berisi paragraf
- Bisa berisi `inline code`

</details>

Kalimat ini memakai <mark>highlight</mark>, lalu juga memuat H<sub>2</sub>O, x<sup>2</sup>, dan tombol <kbd>Ctrl</kbd> + <kbd>C</kbd> untuk melihat apakah elemen HTML sederhana tetap menyatu dengan gaya artikel.

## Escape Character

Kadang kita perlu menampilkan simbol mentah seperti \*asterisk\*, \_underscore\_, dan \#hash tanpa memicu parsing Markdown.
